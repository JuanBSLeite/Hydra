5 What is it?
6 -----------
7 
8 Hydra is a C++11 compliant and header only framework designed to perform commom data analysis tasks on massivelly parallel platforms. Hydra provides a collection of containers and algorithms commomly used in HEP data analysis, which can deploys transparently OpenMP, CUDA and TBB enabled devices, allowing the user to re-use the same code across a large range of available multi-core CPU and accelerators. The framework design is focused on performance and precision.
9 
10 The core algorithms follow as close as is possible the implementations widely used in framworks like ROOT and libraries
11 like GSL.
12 
13 Main features
14 -------------
15 
16 Currently Hydra supports:
17 
18 * Generation of phase-space Monte Carlo samples with any number of particles in the final states. Sequential decays, calculation of integrals of models over the corresponding phase-space and production of weighted and unweighted samples, which can be flat or distributed following a model provided by the user.
19 * Sampling of multidimensional pdfs.
20 * Multidimensional maximum likelihood fits using binned and unbinned data sets.
21 * Calculation of S-Plots, a popular technique for statistical unfolding of populations contributing to a sample.
22 * Evaluation of multidimensional functions over heterogeneos data sets.
23 * Numerical integration of multidimensional functions using self-adaptive Monte Carlo and quadrature methods.
24 * Multidimensional sparse and dense histograming of large samples.
25 
26 Hydra also provides a bunch of custom types, optimized containers and number of algorithms and constructs to maximaze performance, avoiding unecessary usage of memory and without losing the flexibility and protability to compile and run the same code across different platforms and deployment scenarios.
27 
28 For example, just changing .cu to .cpp in any source code writen only using the Hydra and standard C++11 is enough
29 to compile your application for OpenMP or TBB compatible devices using GCC or CLang in a machine without a NVIDIA GPU installed.
30 
31 In summary, using Hydra, the user can transparently typical bottle-neck calculations to a suitable parallel device and get speed-up factors ranging from some dozens to hundeds.

33 Hydra and Thrust
34 ----------------
35 
36 Hydra is implemented on top of the [Thrust library](https://thrust.github.io/) and rely strongly on Thrust's containers, algorithms and backend managment systems.
37 The official version of Thrust supports tuples with maximum ten elements. In order to overcome this limitation, Hydra uses the
38 [unoficial version, forked from the original, by Andrew Currigan and collaborators](https://github.com/andrewcorrigan/thrust-multi-permutation-iterator).
39 This version implements variadic tuples and related classes, as well as provides some additional functionalities, which are missing in the official Thrust.
40 
41 The version of Thrust distributed with Hydra is maintained by [MultithreadCorner](https://github.com/MultithreadCorner). It is basically a fork of Currigan's repository, which was merged with the latest official release available in github (Thrust 1.8.3).
42 
43 ***Hydra does not depend or conflits with the official Thrust library distributed with the CUDA-SDK.***

46 Supported Paralels Backends
47 ---------------------------
48 
49 Hydra uses the underlying Thrust's "backend systems" to control how the algorithms algorithms get
50 mapped to and executed on the parallel processors and accelerators available to a given application.
51 When is necessary, the backends can be specified using the symbols hydra::device::sys_t, hydra::host::sys_t,
52 hydra::omp::sys_t, hydra::tbb::sys_t, hydra::cuda::sys_t, hydra::cpp::sys_t.
53 The backends hydra::device::sys_t and hydra::host::sys_t are selected in compile time using the macros ```THRUST_HOST_SYSTEM``` and ```THRUST_DEVICE_SYSTEM```.
54 The following possibilities are available:
55 
56 * host: THRUST_HOST_SYSTEM_CPP, THRUST_HOST_SYSTEM_OMP, THRUST_HOST_SYSTEM_TBB
57 * device: THRUST_DEVICE_SYSTEM_CPP, THRUST_DEVICE_SYSTEM_OMP, THRUST_DEVICE_SYSTEM_TBB, THRUST_DEVICE_SYSTEM_CUDA
58 
59 For example, this will compile ```my_program.cu``` using OpenMP as host backend and CUDA as device backend:
60 
61 ```bash
62 nvcc -Xcompiler -fopenmp -DTHRUST_HOST_SYSTEM=THRUST_HOST_SYSTEM_OMP -DTHRUST_DEVICE_SYSTEM=THRUST_DEVICE_SYSTEM_CUDA my_program.cu
63 ```
64 The available "host" and "device" backends can be freely combined.
65 Two important features related to the Hydra's design and the backend configuration:
66 
67 * If CUDA backend is not used, [NVCC and the CUDA runtime](https://developer.nvidia.com/cuda-toolkit) are not necessary. The programs can be compiled with GCC, CLang or other host compiler compatible with C++11 directly.
68 * Programs written using only Hydra, Thrust, STL and standard c++ constructs, it means programs without any raw CUDA code or calls to the CUDA runtime API, can be compiled with NVCC, to run on CUDA backends, or a suitable host compiler to run on OpenMP , TBB and CPP backends. **Just change the source file extention from .cu to .cpp, or something else the host compiler understands.**

92 Installation and requirements
93 -----------------------------
94 
95 Hydra is a header only library, so no build process is necessary to install it.
96 Just place the `hydra` folder and its contents where your system can find it.
97 The library run on Linux systems and requires at least a host compiler supporting C++11. To use NVidia's GPUs, CUDA 8 or higher is required.
98 A suite of examples demonstrating the basic features of the library are included in the `examples` folder.
99 All the examples are organized in .inl files, which implements the `main()` function. These files are included by .cpp and .cu
100 files, which are compiled according with the availability of backends. TBB and CUDA backends requires the a installation of the corresponding libraries and runtimes.
101 These code samples uses, but does not requires [ROOT](https://root.cern.ch/) for graphics, and [TCLAP](http://tclap.sourceforge.net/) library for process command line arguments.

103 Examples
104 --------
105 The examples are built using [CMAKE](https://cmake.org/) following the following instructions:
106 
107 1. clone the git repository: `git clone https://github.com/MultithreadCorner/Hydra.git`
108 2. create a build directory: `mkdir build`
109 3. go to build directory: `cd build`
110 4. `cmake ../Hydra`
111 5. `make`
112 
113 
114 The examples are distributed in directories and are named according to the functionalities they ilustrates.
115 The examples are listed below:
116 
117 1. __async__ : async_mc
118 2. __fit__ : basic_fit, multidimensional_fit, extended_logLL_fit, fractional_logLL_fit, phsp_unweighting_functor_and_fit, splot
119 3. __histograming__ : dense_histogram, sparse_histogram
120 4. __misc__ : multiarray_container, multivector_container, variant_types
121 5. __numerical_integration__ : adaptive_gauss_kronrod, gauss_kronrod, plain_mc, vegas
122 6. __phase_space__ : phsp_averaging_functor, phsp_evaluating_functor, phsp_reweighting, phsp_basic, phsp_unweighting, phsp_chain, phsp_unweighting_functor
123 7. __random__ : basic_distributions, sample_distribution
124 
125 Each compiled example executable will have an postfix (ex.: _cuda, _omp, _tbb) to indicate the deployed device backend.
126 All examples use CPP as host backend.
